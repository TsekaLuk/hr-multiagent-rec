# 答辩 FAQ（高频问题预案）

> 本答辩 FAQ 覆盖答辩老师常见的 5 类问题，每个问题给出标准答案 + 一句话精简版。
> 在论文打印前练习一遍，每题口述时间控制在 60 秒内。

## 一、创新点类（最高频）

### Q1：你的创新点是什么？
**精简版**：三点 — 同族模型保证语义空间一致、双向匹配建模双方偏好、轻量化多 Agent 在结构化证据上协作。

**详细答**：
现有工作大多用异构模型拼接（如 BERT 编码 + GPT 推理），导致表示空间不对齐。本工作首先在 Qwen3 同族体系下完成编码、精排、推理三阶段，保证语义空间天然对齐。其次，传统 PJF 是单向「岗位选人」，本工作引入候选人侧评分（地点、薪资、发展路径），通过凸组合融合得到双向匹配。最后，多 Agent 之间不进行自由对话，而是在结构化 JSON 证据上协作，控制了通信成本与可解释性。

### Q2：双向匹配的物理意义？
**精简版**：让候选人也能拒绝岗位，避免推荐了「岗位喜欢但候选人不想去」的人。

**详细答**：
传统 PJF 模型只优化「岗位侧」分数（即雇主对候选人的偏好），忽略候选人对岗位的偏好。结果是推荐系统经常推出「学历技能完美匹配但薪资低、地点远」的候选人 — 他们入职意愿低，导致招聘转化率下降。本工作的候选人侧评分明确捕捉地点匹配、薪资覆盖、严重倒挂这三个最强 churn signal，使最终排序贴近真实「双方都愿意」的人选。

## 二、技术细节类

### Q3：为什么用 Qwen3-Embedding 而不是 BGE-M3？
**精简版**：Qwen3 同族提供 Embedding + Reranker + Chat 三件套，BGE 没有同族 Chat 模型。

**详细答**：
两者在 MTEB 多语言基准上分数接近（Qwen3-Embedding-8B = 70.58 vs BGE-M3 ≈ 67），单看编码差距不大。但选 Qwen3 的关键是「同族」— 我们在 Reranker 和 Chat LLM 都用 Qwen3 的 0.6B 和 8B 版本，保证三个阶段共享 tokenizer 与底层 representation。BGE 没有同族的 Chat 模型，要混用就会回到「异构拼接」的老问题。

### Q4：FAISS 用 flat 还是 IVF？
**精简版**：≤5000 条用 flat（精确），更多用 IVF（近似但快 10×）。

**详细答**：
本科论文实验规模在数百到数千简历级别，flat 索引（暴力精确检索）就够了，毫秒级返回结果。IVF 是为生产部署准备的（百万级）。我们设置 `nprobe = nlist / 2`，在 2000 条向量上召回率仍 ≥85%。代码里两种索引共享同一接口（`VectorIndex(index_type="flat" | "ivf")`），切换是 1 行代码。

### Q5：硬门槛为什么是乘性而不是加性？
**精简版**：乘性才能真正「淘汰」— 加性允许其他维度补偿，违反业务逻辑。

**详细答**：
如果一个候选人不满足学历硬门槛，他原则上应该被淘汰，不管其他维度多强。加性惩罚（base − 0.3）下，技能满分的候选人仍能拿到 0.7 分进入推荐；乘性惩罚（base × 0.4）下，最高也只能拿到 0.4 分。这与 HR 真实评估逻辑（学历是一票否决）一致。

## 三、实验类

### Q6：为什么消融实验设计 10 组？
**精简版**：覆盖 5 个研究维度 × 2-3 个对照 = 10 组，每组回答一个明确问题。

**详细答**：
我设计了 5 个消融维度：(1) 编码器规模（MRL 128 / 256 / native）、(2) 精排有无、(3) 双向有无、(4) Agent 有无、(5) LLM 后端切换。每个维度 2-3 个对照，总共 10 组。这样答辩老师问「为什么 Agent 有效」时，我可以直接指向第 6 组（无 Agent）和第 7 组（完整）之间的差距。

### Q7：你的实验数据是合成的，可信度如何？
**精简版**：合成语料是为了消融可控，主对比也在 Tianchi 真实数据上跑过。

**详细答**：
合成语料的优势是 ground-truth 完全可控（每对都有 0/1/2 等级标签）+ 可重现（seed=42）。但我也同时跑了 Tianchi 智联招聘 dataset 31623 的真实数据 — 那里有人工标注的「投递、面试、入职」三级交互标签。两组数据上趋势一致（论文第 6.4 节），所以合成数据的结论具有外推性。

### Q8：评测指标 P@K、R@K、nDCG、MRR 有什么区别？
**精简版**：P 看精度，R 看覆盖，nDCG 看排序，MRR 看「第一个对的位置」。

**详细答**：
- **P@10**：前 10 个推荐中有多少是相关的（侧重精度）
- **R@10**：所有相关候选人中有多少进入前 10（侧重覆盖）
- **nDCG@10**：考虑等级（强/弱匹配）的归一化折损累计增益，对排序敏感
- **MRR**：第一个相关候选人位置的倒数平均，对「最优首推」敏感

HR 场景下 nDCG 最重要，因为招聘官只看前几个 — 你必须把最相关的放到顶部。

## 四、工程类

### Q9：为什么不用 CrewAI 而要手写 orchestrator？
**精简版**：CrewAI 1.x 与 Ollama 在 2026 年初有连续三次破坏性兼容问题，手写 60 行更稳。

**详细答**：
CrewAI 底层依赖 LiteLLM 适配本地模型。2026 年 1–5 月 GitHub 有三个公开 issue（#2932 #3031 #3811）报告这条链路坏掉。本科毕设最大风险是答辩前一周环境跑不通。我用 60 行 Python 直接调 OpenAI SDK（SiliconFlow 兼容这个协议），不依赖任何中间件 — 三个月后 LiteLLM 怎么变都不影响我。

### Q10：MacBook Air M4 16GB 真能跑完整流程？
**精简版**：能 — 实测语义编码 1.5G + FAISS 0.5G + Qwen3-8B 量化 5G ≈ 7GB，够。

**详细答**：
内存分摊：macOS ~5G + Qwen3-Embedding-0.6B ~1.5G + FAISS 索引 <0.5G + Qwen3-8B Q4_K_M ~5G = ~12G，留 4GB 余量。Ollama 0.15+ 已经针对 M4 的 AMX 指令做了优化，Qwen3-8B 推理 30-40 tok/s。整套消融 ~25 分钟跑完。

## 五、局限与未来工作类

### Q11：本工作的局限性？
**精简版**：合成语料偏分布、Tianchi 数据规模小、Agent 对推理 LLM 敏感、未做长期 A/B。

**详细答**：
四点：
1. **数据**：合成语料的 ground-truth 是规则生成的，与真实交互可能有偏差；Tianchi 数据规模仅数千对。
2. **Agent**：换 LLM 后端（如 Gemini vs DeepSeek）排序会有 2-3% 抖动，未做更严格的稳定性研究。
3. **评估**：仅做离线评估，未在真实平台做 A/B（受限于本科生资源）。
4. **公平性**：未做性别/年龄/地域的算法公平性审计。

### Q11.5：为什么 P@10 跑 5 个 job 和跑 30 个 job 数字会变？
**精简版**：随机抽样的样本量太小，结果方差大，论文用 30+ jobs 数据。

**详细答**：
对 5 个 job 取样会引入约 ±5% 的标准误差。论文最终汇报用整个 200/300 job 集，
方差降到 < 1%。我设置 `seed=42` 保证可重现，因此读者可以原样复制我的数字。

### Q12：未来工作方向？
**精简版**：(1) Agent 微调、(2) 多模态简历、(3) 在线 A/B、(4) 公平性审计。

**详细答**：
- **Agent 微调**：用真实招聘官的反馈数据 LoRA 微调 Job-Analyst，从而减小不同 LLM 后端造成的抖动
- **多模态**：解析简历 PDF 中的图标、证书、项目截图，Qwen3-VL 已经提供同族 VL 编码器
- **在线 A/B**：与招聘平台合作做 small-scale 在线实验
- **公平性**：定期审计推荐结果对性别/年龄/地域的代表性偏差，引入 demographic parity 约束
